\begin{frame}
\frametitle{$p_T(B)$--issue: sWeights and gb-weights}
Given the machinery we got from the previous round of the analysis, to compute
the sWeights, there are 3 consistent ways of using the gb-weights:
\begin{enumerate}
  \item<2-> Not using gb-weights. \only<2>{Then the MC prefit (cats 0 and 50 only)
    for the $B_s^0$ RD mass fit does not know about the tails in RD.}
    \only<3->{$\rightarrow\, p_T$--issue}
  \item<4-> Using gb-weights for the tails to compute \texttt{sw}, but 
    \emph{(almost) not} in the acceptances. \only<4>{This is what was done in
    the previous paper. Briefly, we achieved this setting \texttt{sw' = sw/gbw}
    for $B_s^0$ MC's and \texttt{sw' = sw} for $B_d^0$ MC. Then during
    acceptances we were using \texttt{sw'}.}
    \only<5->{$\rightarrow\, p_T$--issue}
  \item<6-> Using gb-weights in tails and acceptances. \only<6>{This was first
    seen in December 2021, and after some discussion in during the workshop 
    in Santiago.}
    \only<7->{$\rightarrow\, \textbf{no}\, p_T$--issue}
\end{enumerate}

\begin{frame}
\frametitle{Ana-preservation--issue}

Since the multithreading was enabled, then each time we ran the pipeline, the
events were shuffled. There are 3 parts of the analysis which would give
different results in this situation:
\begin{itemize}
  \item \textbf{PID gen for the muons.} \only<2>{I checked these distributions when
    just shuffling the events before and they look very similar. It is hard 
    believe the change in PID would really affect the final result so much.
    Anyway, \emph{we will assess it in the systematics.}
  }
  \item \textbf{GB weights when using 60\% of the sample.} \only<3>{If we do not use
    the whole dataset it is obvious the distribution-matching is going to be
    worse. Also in 2016, we were using the reweigther trained on the 60\% of
    just on sim version for the other one (could introduce also differences).
    \emph{We can assess this just by shuffling the events for two tuples with the
    same PID on run1-cut--based tuples.}}
  \item \textbf{BDT selection.} \only<4>{Both the two previous differences add up here,
    since GBW are quite important in the BDT. Also the BDT itself is a bit
    sensible. \emph{To assess this we just need to shuffle after the GBW.}}
\end{itemize}

\bigskip

\only<5->{To fix/understand this...}
\begin{itemize}
  \item<5->{Just by disabling multithreading we would have reproducible results.
  But is not enough.}
  \item<6->{Once we fix the selection, we can run 3 set of tuples and know what is the major
source of difference. Most probably, it is the gb-weigthing, which also enters
in the sweighing procedure, though.}
\end{itemize}

\end{frame}

pipeline and fully integrated it into \texttt{phis-scq/selection}. To speed it
up I prepare it to run in HTCondor and made some other changes:
\begin{itemize}
  \item Disable multithreading.
  \item Change gb-weight subsampling to 1.
  \item Move backgorund subtraction (before GBW) to double-sided CrystalBall.
